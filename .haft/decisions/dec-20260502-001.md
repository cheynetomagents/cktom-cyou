# Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu)

## 1. Problem Frame

**Signal:** Current Proxmox+OpenTofu baseline is narrow and tool-specific. Real goal: single declarative source of truth covering bare metal, hypervisors, VMs (with cloud-init), OCI containers, networking, secrets, storage — where an LLM generates PRs and merging triggers automated deployment. OpenTofu was first candidate but may not be best fit. Kubernetes/KubeVirt, NixOS, Incus, and other stacks are viable alternatives not yet evaluated.

**Constraints:**
- PCIe passthrough (VFIO/IOMMU) must work — currently 2 GPUs on node1, more planned
- Must support OCI containers as primary workload unit
- All config in git — no manual state outside repo
- Secrets never plaintext in repo
- Solo operator — ops complexity matters
- LLM PR-gen pipeline is out of scope for now — tool must expose machine-readable, LLM-friendly config format

**Acceptance:** Single declarative repo drives all layers: bare metal config, hypervisor/VM provisioning, cloud-init, OCI workloads, networking, secrets, storage. Merging a PR to describe new software deployment causes it to appear in the environment. GPU passthrough works. Solo operator can understand and debug the full stack.

## 2. Decision

**Selected:** Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu)

**Selection policy:** Maximize declarative_coverage and minimize ops_complexity for solo operator; observability_integration as tiebreaker. All three constraints (GPU passthrough, OCI native, secrets-safe gitops) must pass.

**Why selected:** Solo operator, Proxmox familiarity retained, 75% OCI workload gets native ArgoCD GitOps immediately, best observability coverage (Proxmox collector + k8s daemonset both layers), migration cost is medium not high. Two-plane coordination overhead is acceptable tradeoff. Stepping stone to full k8s+KubeVirt if desired later.


**Invariants:**
- GPU passthrough (VFIO/IOMMU) must work for vLLM and llama.cpp VMs on node1
- No plaintext secrets in any tracked file
- All config changes flow through git PR — no manual API or UI changes that bypass git
- Proxmox VM layer (appliances) managed by OpenTofu; OCI container layer managed by ArgoCD/k3s
- tofu plan must be clean before any VM-layer PR merges

**Pre-conditions:**
- [ ] Existing node3/node4 OpenTofu configs retained and extended to node1
- [ ] k3s cluster provisioned (can run as Proxmox VMs on node3/node4)
- [ ] ArgoCD installed in k3s cluster
- [ ] Secret backend chosen and configured (SOPS or Vault) before any workload migration
- [ ] Netdata running on all Proxmox nodes

**Post-conditions:**
- [ ] tofu plan exits 0 across node1/node3/node4 workspaces
- [ ] ArgoCD syncs cleanly from git repo
- [ ] GPU passthrough working for vLLM/llama.cpp VMs
- [ ] Netdata shows metrics from both Proxmox collector and k8s daemonset
- [ ] No secrets in git history
- [ ] New software deployable by adding k8s manifest or Tofu resource to repo and merging PR

**Admissibility:**
- NOT: No manual Proxmox UI changes that create resources outside Tofu state
- NOT: No kubectl apply outside ArgoCD sync path
- NOT: No plaintext credentials in any .tf, .yaml, or .env file tracked by git
- NOT: No VM workloads migrated to k8s/KubeVirt without explicit re-evaluation of this decision

## 3. Rationale

**Counterargument:** V3 (k8s+KubeVirt) is the architecturally cleaner long-term answer — single control plane, 86% declarative coverage, structured LLM surface. The two-plane approach in V4 is a pragmatic compromise that may require rework if KubeVirt matures and GPU passthrough via KubeVirt becomes reliable.

**Selected variant weakest link:** Two declarative planes (Tofu for VMs, ArgoCD/k3s for containers) — LLM PR-gen must know which plane each resource targets. Coordination boundary between VM layer and container layer is a hidden complexity sink; VM-to-pod dependency ordering not enforced by either plane.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu) | **Selected** | Solo operator, Proxmox familiarity retained, 75% OCI work... |
| V3: k8s+KubeVirt+ArgoCD | Rejected | ops_complexity=high and migration_effort=high wrong risk profile for solo operator bootstrapping from Proxmox. Needs 3rd node for HA control plane. KubeVirt appliance VM management (Home Assistant, Security Onion) significantly more complex than Proxmox VMs. |
| V1: Proxmox+OpenTofu | Rejected | No native OCI workload support (LXC wrappers only) and no native GitOps loop. V4 strictly better on target dimensions. |
| V2: Proxmox+Pulumi | Rejected | Same coverage ceiling as V1 (57%), community provider risk, Python LLM surface advantage is observation-only dimension. V4 beats on all targets. |

**Evidence requirements:**
- tofu plan output showing node1 GPU VM with PCIe passthrough
- ArgoCD sync showing clean application from git repo
- Netdata dashboard showing both Proxmox and k8s metrics
- git log showing no secrets committed

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| ArgoCD GitOps loop covers 75% OCI workload without manual intervention | New container deployment merged via PR appears in k3s cluster within ArgoCD sync window | < 5 minutes from merge to running pod |
| Proxmox OpenTofu covers GPU VM provisioning declaratively | tofu plan shows vLLM/llama.cpp VMs with correct PCIe passthrough config, exits 0 | 0 plan errors, PCIe device present in VM config |
| Two-plane ops complexity stays manageable for solo operator | Operator can debug a failed deployment (container or VM) without consulting more than 2 tools | ArgoCD UI + kubectl OR Proxmox UI + tofu plan — not both simultaneously |

## 4. Consequences

**Rollback plan:**
Triggers:
- k3s cluster instability causes OCI workload downtime > 4 hours
- ArgoCD sync loop causes runaway apply cycles
- Tofu state corruption on VM layer
Steps:
1. Disable ArgoCD auto-sync, switch to manual sync mode
2. Roll back affected k8s manifests via git revert + manual ArgoCD sync
3. For VM layer: tofu state rm affected resources, re-import, re-apply
4. If full rollback needed: migrate OCI workloads back to Docker-in-LXC on Proxmox (V1 baseline)
Blast radius: Container workloads only for k3s issues; VM workloads only for Tofu issues. Planes are independent — failure in one does not cascade to other.

**Refresh triggers:**
- KubeVirt reaches CNCF graduation and GPU passthrough declared stable
- Second operator joins — two-plane model becomes coordination problem
- ops_complexity exceeds solo-maintainable threshold (3+ incidents requiring both planes simultaneously)
- Proxmox bpg/proxmox provider 1.0 rewrite introduces breaking changes requiring re-evaluation

**Affected files:** providers.tf, variables.tf, node3.tf, node4.tf, node1.tf, networking.tf, pools.tf, imports.tf
