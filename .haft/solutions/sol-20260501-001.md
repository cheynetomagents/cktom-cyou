# Solutions for: Choose declarative homelab management stack for LLM-driven GitOps workflow

Problem: prob-20260501-001

## Variants (4)

### V1. Proxmox + OpenTofu (current baseline, extended)

Keep Proxmox as hypervisor, extend current bpg/proxmox OpenTofu config to cover node1, LXCs, OPNsense, Zyxel (Ansible). Add GitHub Actions for tofu plan/apply on PR merge to close GitOps loop. Secrets via external backend (Vault or SOPS).

**Strengths:**
- Already 70% done — node3/node4 configs exist, PCIe passthrough works
- bpg/proxmox provider has mature GPU/VFIO support via hardware_mappings
- Netdata has deep native Proxmox collector — zero-config cgroup/VM metrics
- Lowest migration effort — no hypervisor change

**Novelty:** Incremental extension of existing work. Only variant with zero hypervisor change.

**Weakest link:** GitOps loop requires custom GitHub Actions glue — no auto-sync operator. LLM-generated HCL must pass manual tofu plan before merge; no CRD watch loop means drift can accumulate silently between applies.

**Risks:**
- bpg/proxmox SDKv2 provider in maintenance mode — 1.0 rewrite in progress, potential breaking changes
- State file is plaintext unless remote backend configured — secrets leak risk
- Proxmox HA migration causes drift — provider doesn't know if VM moved nodes

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Proxmox stays as hypervisor; could layer ArgoCD for OCI workloads later, or migrate to Pulumi without touching VMs.

**Rollback:** No hypervisor change — rollback is revert git commit + tofu apply. Low blast radius.

### V2. Proxmox + Pulumi (Python, Automation API)

Keep Proxmox as hypervisor, replace OpenTofu with Pulumi Python SDK. Pulumi Automation API enables programmatic apply from CI without manual tofu commands. Python surface is more LLM-friendly than HCL. Pulumi ESC handles secrets. Same Proxmox API underneath.

**Strengths:**
- Python DSL: loops, functions, conditionals — LLM generates fewer syntax errors than HCL
- Pulumi Automation API: trigger apply programmatically on PR merge without shell scripts
- Pulumi ESC: purpose-built secrets management with Vault/1Password/AWS integration
- Same Proxmox API as current setup — PCIe passthrough and Netdata unchanged

**Novelty:** Only variant that swaps IaC tool while keeping hypervisor. Targets LLM codegen surface specifically.

**Weakest link:** Pulumi proxmoxve provider is community-maintained (not official Pulumi org) — less battle-tested than bpg/proxmox. LLM-generated Python can introduce unsafe patterns (shell injection, secrets in logs) more easily than declarative HCL.

**Risks:**
- Community provider maturity risk — ryan4yin/pulumi-proxmoxve may lag behind Proxmox API changes
- Pulumi ESC secrets sharing requires paid Pulumi Cloud for team/org features
- Python imperative surface means LLM can generate valid-but-wrong logic (loops, conditionals) that HCL would reject at parse time

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Python codebase can grow into full Pulumi Automation API pipeline; same Proxmox base means VM configs portable.

**Rollback:** Proxmox unchanged — rollback is revert to OpenTofu configs. State migration from Tofu to Pulumi is one-way without tooling.

### V3. Kubernetes + KubeVirt + ArgoCD (k8s-native everything)

Run k8s cluster (3 nodes via kubeadm or k3s) on bare metal or as Proxmox VMs. KubeVirt manages appliance VMs as CRs. ArgoCD watches git repo and auto-applies on merge. All workloads — containers and VMs — defined as YAML CRs. GPU passthrough via KubeVirt permittedHostDevices.

**Strengths:**
- ArgoCD GitOps loop is native — merge PR = auto-deploy, no glue scripts
- YAML CRs are maximally LLM-friendly — schema-driven, kubectl-validated
- External Secrets Operator + RBAC: best-in-class secrets management in this set
- Netdata k8s daemonset: per-pod cgroup metrics, Prometheus scrape, strong integration

**Novelty:** Only variant where VMs and containers share a single control plane (k8s). Fundamentally different operational model.

**Weakest link:** KubeVirt VM management is significantly more complex than Proxmox for appliances (Home Assistant, Security Onion) — these were designed for direct hypervisor access, not k8s VM CRs. KubeVirt CNCF graduation pending; ecosystem still maturing.

**Risks:**
- Minimum 3-node cluster for HA control plane — your 2-node Proxmox needs a 3rd VM or bare metal node
- Network overlay (Flannel/Calico) adds latency vs Proxmox SDN; Security Onion performance-sensitive
- PCIe passthrough via KubeVirt requires node-level VFIO config + KubeVirt CR — more moving parts than Proxmox hardware_mappings

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Full k8s foundation enables future: Knative for serverless, Crossplane for cloud resources, Cluster API for multi-cluster. Maximum future optionality.

**Rollback:** High blast radius — full hypervisor migration. Rollback requires rebuilding Proxmox environment. Keep Proxmox nodes available during migration window.

### V4. Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu)

Keep Proxmox as hypervisor for VMs (appliances + GPU workloads). Run a lightweight k8s cluster (k3s) inside Proxmox VMs for the 75% OCI container workload. ArgoCD manages containers. OpenTofu manages VM layer. Two declarative planes that don't interfere.

**Strengths:**
- Best-fit tool per layer: Tofu for VMs (mature GPU passthrough), ArgoCD/k8s for containers (native GitOps)
- ArgoCD auto-sync on merge for container workloads — covers 75% of the use case immediately
- No KubeVirt complexity — VMs stay as Proxmox VMs, containers stay as pods
- Netdata covers both layers: Proxmox collector for VMs, k8s daemonset for pods

**Novelty:** Only variant with two independent declarative planes — pragmatic split by workload type rather than unified control plane.

**Weakest link:** Two declarative planes = two mental models, two state stores, two failure domains. LLM must know which plane to target for each resource type. Coordination boundary between Tofu VM layer and k8s app layer is a hidden complexity sink.

**Risks:**
- Cognitive overhead: solo operator debugging spans Proxmox UI, tofu state, k8s events, ArgoCD sync — 4 separate tools
- VM-to-container dependency ordering not enforced — k8s pod can start before backing VM is ready
- k3s inside Proxmox VMs adds nesting overhead — storage/network performance degrades

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** k3s cluster can grow into full k8s + KubeVirt (V3) later if VM management via Tofu proves limiting. Proxmox stays as stable base.

**Rollback:** Container layer rollback via ArgoCD git revert. VM layer rollback via tofu revert. Independent — but coordination bugs require coordinated rollback.

## Dominated Variant Elimination

- Proxmox + OpenTofu (current baseline, extended) — dominated by Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu). V4 matches or beats V1 on all target dimensions — adds native OCI + ArgoCD GitOps with marginal ops_complexity increase. V1 has no compensating win.
- Proxmox + Pulumi (Python, Automation API) — dominated by Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu). V2's unique strength (structured Python LLM surface) is an observation dimension only. V4 beats V2 on declarative_coverage and oci_primary_workload with comparable ops complexity. Community provider risk uncompensated.

## Pareto Front Trade-Offs

- Proxmox + OpenTofu (current baseline, extended) — Dominated by V4. Lowest migration effort and ops complexity, but OCI is second-class (LXC wrappers) and no native GitOps loop.
- Proxmox + Pulumi (Python, Automation API) — Dominated by V4. Best LLM codegen surface (Python DSL), but same coverage ceiling as V1 with added provider risk.
- Kubernetes + KubeVirt + ArgoCD (k8s-native everything) — Best declarative coverage (86%), native OCI, native GitOps, structured LLM surface. Sacrifices ops_complexity (high) and migration_effort (high). Right if k8s investment is acceptable.
- Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu) — Best observability, native OCI for 75% workload, ArgoCD for containers, Proxmox familiarity for VMs. Sacrifices unified control plane and LLM codegen surface (two schemas). Right if minimizing bootstrap risk matters more.

**Selection policy:** Maximize declarative_coverage and minimize ops_complexity; observability_integration as tiebreaker. Constraints (gpu_passthrough_support, oci_primary_workload, gitops_secrets_safe) must all pass.

**Recommendation (advisory):** Proxmox + ArgoCD hybrid (containers via k8s, VMs via Tofu)

**Recommendation rationale:** V4 recommended for solo Proxmox operator: ops_complexity=med beats V3's high, observability best across both layers, 75% OCI gets native GitOps immediately, migration_effort=med vs V3's high. Coverage gap (71% vs 86%) is a known coordination tradeoff, not a structural ceiling. V4 is also a stepping stone to V3 if k8s comfort grows.
