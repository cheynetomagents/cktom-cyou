# Choose declarative homelab management stack for LLM-driven GitOps workflow

## Signal

Current Proxmox+OpenTofu baseline is narrow and tool-specific. Real goal: single declarative source of truth covering bare metal, hypervisors, VMs (with cloud-init), OCI containers, networking, secrets, storage — where an LLM generates PRs and merging triggers automated deployment. OpenTofu was first candidate but may not be best fit. Kubernetes/KubeVirt, NixOS, Incus, and other stacks are viable alternatives not yet evaluated.

## Constraints

- PCIe passthrough (VFIO/IOMMU) must work — currently 2 GPUs on node1, more planned
- Must support OCI containers as primary workload unit
- All config in git — no manual state outside repo
- Secrets never plaintext in repo
- Solo operator — ops complexity matters
- LLM PR-gen pipeline is out of scope for now — tool must expose machine-readable, LLM-friendly config format

## Optimization Targets

- Declarative coverage breadth — % of infra surface (bare metal, VMs, containers, networking, storage, secrets) expressible as code
- Ops complexity — time/effort for solo operator to maintain, debug, and extend
- LLM codegen surface quality — how structured/predictable the config DSL is for automated PR generation

## Observation Indicators

- Total migration effort from current Proxmox baseline
- Community ecosystem size (docs, plugins, tooling)
- Hardware resource overhead of control plane

## Acceptance

Single declarative repo drives all layers: bare metal config, hypervisor/VM provisioning, cloud-init, OCI workloads, networking, secrets, storage. Merging a PR to describe new software deployment causes it to appear in the environment. GPU passthrough works. Solo operator can understand and debug the full stack.

## Blast Radius

Entire homelab — node1 (2 GPUs), node3, node4, router1 (OPNsense), Zyxel switch. All running workloads.

## Reversibility

low

| Dimension | Role | Scale | Unit | Polarity | Measurement |
|-----------|------|-------|------|----------|-------------|
| declarative_coverage | target | ratio | % | higher_better | Count infra surface areas (bare metal config, VM provisioning, cloud-init, OCI workloads, networking, secrets, storage) natively declarable without manual steps. Divide by 7. Score each area: 1=fully declarative, 0.5=partial/workaround, 0=manual only. |
| ops_complexity | target | ordinal | low/med/high | lower_better | Estimate: (1) time to onboard fresh clone to working state, (2) number of distinct tools/agents a solo operator must understand, (3) blast radius of a single misconfiguration. Aggregate to ordinal. |
| observability_integration | target | ordinal | native/plugin/manual | higher_better | Does Netdata have a first-party integration? Plugin ecosystem? Or requires custom exporters/scrape configs only? |
| gpu_passthrough_support | target | nominal | supported/workaround/unsupported | higher_better | Can VFIO/IOMMU PCIe passthrough be configured declaratively for VMs? Must = supported. |
| oci_primary_workload | target | nominal | native/supported/unsupported | higher_better | OCI containers must be first-class workload unit, not bolted on. Must = native or supported. |
| gitops_secrets_safe | target | nominal | yes/no | higher_better | Can all config live in git with secrets managed via external secret store (no plaintext in repo)? Must = yes. |
| llm_codegen_surface | target | ordinal | structured/mixed/freeform | higher_better | Is the config DSL schema-driven and deterministic enough for LLM PR generation? Structured=schema+validation, mixed=partially, freeform=prose-like. |

| community_ecosystem | target | ordinal | large/medium/small | higher_better | GitHub stars, active maintainers, plugin count, SO/forum activity, docs quality. |

**Parity rules:** All variants evaluated against same target inventory: node1 (2 GPUs), node3, node4, router1 (OPNsense), Zyxel XMG1915-10E. Same workload mix: 75% OCI containers, 25% VMs (appliances). Same secret backend assumption (external store, TBD). Same operator profile: solo. Same Netdata observability requirement.
